# Decoder Conformance

This document defines the conformance target for the first Uya Opus release.
Decoder conformance has priority over encoder quality.

## Output Rules

Bit-exact output is required when the reference behavior is fully specified by
RFC 6716/RFC 8251 and uses fixed-point decoder paths. A decoded PCM frame is
bit-exact only when every interleaved `i16` sample matches the reference output
for the same packet sequence, sample rate, channel count, PLC/FEC mode, and
pre-skip handling.

Error-tolerant comparison is allowed only when a reference source documents a
permitted tolerance or when container resampling/dither outside the codec core
is explicitly part of the comparison. Any tolerance must be recorded with:

- reference source;
- sample format;
- maximum absolute error;
- maximum accumulated error or SNR threshold;
- reason bit-exact comparison is not applicable.

The packet parser, entropy decoder, and fixed-point codec core must not use
tolerance-based checks for integer state transitions.

## Invalid Input Rules

All byte-slice packet inputs must either decode successfully or return a stable
Uya error. Invalid, truncated, overlong, non-canonical, or over-duration packets
must not crash, read out of bounds, allocate unbounded memory, or leave decoder
state partially advanced after a reported error.

## Corpus Sources

Primary corpus sources:

- RFC 6716 Appendix A/B material and examples where applicable.
- Official Opus test vectors distributed by Xiph.Org.
- Packets generated by the fixed upstream baseline in `docs/references.md`.
- Project-local regression vectors under `tests/vectors/`.

Current project-local SILK-only coverage is locked by deterministic mono
vectors in `tests/silk_decoder.uya`. These vectors cover NB/MB/WB packets at
10/20/40/60 ms and compare a fixed decoded-output checksum matrix. They are a
regression gate for the in-repository decoder path; they do not replace the
external Xiph.Org corpus required by the release conformance gate below.

Differential references may use external tools such as `opus_demo`, `opusdec`,
or FFmpeg only from `tools/` or `tests/`. They must not become runtime
dependencies of `src/opus/**`.

## Acceptance Commands

At the empty-skeleton stage, conformance documentation is accepted by:

```sh
rg -n 'Bit-exact|Corpus Sources|Acceptance Commands|maximum absolute error' docs/conformance.md
git diff --check -- docs/conformance.md docs/todo.md
```

After the build skeleton exists, the project-level conformance gate becomes:

```sh
make check
make test
```

After decoder corpus tooling exists, the release conformance gate becomes:

```sh
make test
tools/diff_decode.sh tests/vectors
```

The release gate is not satisfied until all required corpus failures are either
fixed or documented as blocked with a concrete upstream/toolchain reason.
